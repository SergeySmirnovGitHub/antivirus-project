## Этап 2 — мульти-источник: локальные движки + несколько баз
Цель: вердикт не по одному VirusTotal, а агрегированный по нескольким независимым
источникам — так точнее и устойчивее к пропускам отдельного движка.

> ⚠️ КРИТЕРИЙ (не нарушать): всё работает из ОДНОЙ папки. Пользователь скачивает
> одну нашу программу — и она работает. ClamAV и др. локальные движки **бундлятся
> или авто-провижнятся в подпапку рядом с программой** (`<папка_exe>/engines/...`),
> НЕ ставятся отдельно и НЕ разбросаны по системе. Базы качаем в папку приложения.
> Подробнее: память `single-folder-constraint`.

- [x] Общий интерфейс «движок проверки» (`engines.py`: EngineResult) + агрегатор вердиктов
- [x] VirusTotal как источник в разбивке (`_vt_engine_result`); логика VT в `_scan_virustotal`
- [x] ClamAV-движок (обёртка над `clamscan`, `engines.py`) — протестирован на Mac имитацией
- [x] Разбивка вердикта по источникам в выводе (├ VirusTotal ● … └ ClamAV ● …)
- [x] Тест на EICAR: VirusTotal + ClamAV(stub) → агрегированный вердикт ВРЕДОНОСНЫЙ
- [x] **Загрузчик ClamAV** `provision_clamav()` + команда `setup-clamav` (CLI и GUI):
      качает портативный ClamAV (clamav.net) в `data_dir/engines/clamav` + freshclam,
      рекурсивный поиск clamscan. Своя папка создаётся сама. ⚠️ Тестировать на Windows.
- [~] (старый пункт) авто-провижн ClamAV в папку приложения (команда `setup-clamav`): скачать
      портативный clamscan для Windows + базу сигнатур в `<папка_exe>/engines/clamav/`.
      Критерий «одна папка». Тестировать на Windows (нет доступа с Mac).
- [ ] YARA — локальные правила (эвристика по содержимому файла), база в папке приложения
- [ ] Офлайн-режим: базовый вердикт без интернета (ClamAV + YARA)

### Открытые базы/источники для подключения (бесплатные)
- [ ] MalwareBazaar (abuse.ch) — поиск по SHA-256, бесплатный API-ключ
- [ ] MetaDefender Cloud (OPSWAT) — мультидвижковая проверка по хэшу, бесплатный тариф
- [ ] MalShare — база хэшей вредоносов, бесплатный API-ключ
- [ ] Hybrid Analysis — репутация/анализ по хэшу, бесплатный API-ключ
- [ ] AlienVault OTX — Open Threat Exchange, репутация/IOC, бесплатно
- [ ] ThreatFox (abuse.ch) — индикаторы компрометации по хэшу
- [ ] **Kaspersky OpenTIP** — `GET https://opentip.kaspersky.com/api/v1/search/hash?request=<hash>`,
      нужен бесплатный API-токен (как ключ VT). Поддержка MD5/SHA1/SHA256, есть квоты (403 при
      превышении). Реально полезный источник; есть офиц. KasperskyLab/OpenTIP-scanner на GitHub.
      Добавить движком (как ClamAV/VT) — пользователь получает свой токен.
- [ ] NSRL (whitelist) — база известных «хороших» хэшей, гасит ложные срабатывания
> У каждого источника свои лимиты и формат ответа. Агрегатор приводит их к единому
> виду и считает итог, напр.: «N из M источников считают файл опасным».

### Фоновое наблюдение — ядро готово (`monitor.py`), команда CLI `monitor`

> срабатывания нельзя крушить автоматически). Жёсткое удаление — вручную/по подтверждению.
- [x] `watchdog` — слежение за новыми файлами (Downloads/Desktop) → автоскан → карантин
- [x] `psutil` — снимок процессов + диф (новые процессы), пауза/возобновление
- [x] Монитор автозагрузки ⭐ (реестр Run/RunOnce + папка Startup; снимок+диф → алерт)
- [x] Простая эвристика: запуск из Temp/Downloads → warn
- [x] Тест на Mac: новый файл → скан → авто-карантин; снимки/диф/suspend
- [ ] Проверить на Windows: `vtscan-cli.exe --monitor` (реестр, процессы, карантин)
- [ ] Интеграция мониторинга в GUI-окно (события в ленте, тумблер «Защита вкл»)
- [ ] «Сканировать компьютер» (быстрый/полный скан) — рекурсивный обход; полный скан
      реалистичен с ClamAV (VT упирается в лимит 4/мин). Кнопка в GUI.
- [ ] Упаковка как автозапускаемый фоновый сервис / трей

## Этап 3.5 — download-gatekeeper (идея пользователя, разнесена по этапам)
> Хотим: скачиваемый файл проверяется ДО завершения загрузки, и докачивается/
> становится доступным только после одобрения. Три пути реализации:
> 1. **Расширение браузера** (downloads API) — пауза загрузки до вердикта. User-mode,
>    реалистично, но только для поддержанных браузеров. → можно сделать в рамках этапа 3.
> 2. **Локальный MITM-прокси** — скан файла до отдачи браузеру. Требует вскрытия HTTPS
>    и своего сертификата. Тяжело, для учебного этапа избыточно. → отложено.
> 3. **Minifilter-драйвер** — настоящая блокировка открытия/записи файла для любой
>    программы в реальном времени. → это этап 4 (см. ниже).

## Фишки безопасности (чем выделиться) — ✅ одобрено, разложено по этапам
> Почти всё одобрено пользователем. Метка [этап N] показывает, куда ложится.
> Большинство — фоновый режим (этап 3), user-mode, без драйвера ядра.

- [ ] **Монитор автозапуска (persistence)** [этап 3] — снимок Run-ключей реестра,
      папки «Автозагрузка», Планировщика, Служб → алерт при появлении нового
      (там прячется ~90% малвари). ⭐ ПРИОРИТЕТ. Смыкается с мониторингом процессов.
- [ ] **Ловушка от шифровальщиков (canary)** [этап 3] — несколько маленьких файлов-
      приманок на папку (килобайты, не пустые), процесс начал их массово менять →
      заморозить. Нагрузки на диск/память нет. Реальная защита от ransomware (`watchdog`).
- [ ] **Поведенческая эвристика процессов** [этап 3] — следим за поведением, а не файлом:
      документ запускает PowerShell/cmd, exe из Temp/Downloads лезет в сеть, неподписанный
      плодит процессы → «очки подозрительности» → алерт/заморозка. Лёгкая (`psutil`).
- [ ] Авто-скан флешек/USB при подключении [этап 3].
- [ ] Проверка ссылок (анти-фишинг) по URLhaus/PhishTank — до клика [этап 3].
- [ ] Монитор сетевых подключений + подсветка «плохих» адресов (ThreatFox) [этап 3].
- [ ] Контроль целостности критичных файлов (tripwire): `hosts`, DNS, системные [этап 3].
- [ ] **«Семейный режим»** — отчёты человеческим языком + авто-сканы по расписанию [этап 3].
- [ ] Приватность как фича: всё локально, наружу только хэши в выбранные базы [принцип].

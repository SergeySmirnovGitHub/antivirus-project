## Этап 2 — мульти-источник: локальные движки + несколько баз
Цель: вердикт не по одному VirusTotal, а агрегированный по нескольким независимым
источникам — так точнее и устойчивее к пропускам отдельного движка.

> ⚠️ КРИТЕРИЙ (не нарушать): всё работает из ОДНОЙ папки. Пользователь скачивает
> одну нашу программу — и она работает. ClamAV и др. локальные движки **бундлятся
> или авто-провижнятся в подпапку рядом с программой** (`<папка_exe>/engines/...`),
> НЕ ставятся отдельно и НЕ разбросаны по системе. Базы качаем в папку приложения.
> Подробнее: память `single-folder-constraint`.

- [x] Общий интерфейс «движок проверки» (`engines.py`: EngineResult) + агрегатор вердиктов
- [x] VirusTotal как источник в разбивке (`_vt_engine_result`); логика VT в `_scan_virustotal`
- [x] ClamAV-движок (обёртка над `clamscan`, `engines.py`) — протестирован на Mac имитацией
- [x] Разбивка вердикта по источникам в выводе (├ VirusTotal ● … └ ClamAV ● …)
- [x] Тест на EICAR: VirusTotal + ClamAV(stub) → агрегированный вердикт ВРЕДОНОСНЫЙ
- [x] **Загрузчик ClamAV** `provision_clamav()` + команда `setup-clamav` (CLI и GUI):
      качает портативный ClamAV (clamav.net) в `data_dir/engines/clamav` + freshclam,
      рекурсивный поиск clamscan. Своя папка создаётся сама. ⚠️ Тестировать на Windows.
- [~] (старый пункт) авто-провижн ClamAV в папку приложения (команда `setup-clamav`): скачать
      портативный clamscan для Windows + базу сигнатур в `<папка_exe>/engines/clamav/`.
      Критерий «одна папка». Тестировать на Windows (нет доступа с Mac).
- [ ] YARA — локальные правила (эвристика по содержимому файла), база в папке приложения
- [ ] Офлайн-режим: базовый вердикт без интернета (ClamAV + YARA)

### Фоновое наблюдение — ядро готово (`monitor.py`), команда CLI `monitor`
> Решение по безопасности: реакция по умолчанию ОБРАТИМАЯ — карантин файла (move в
> `data_dir/quarantine`) и пауза процесса (suspend), а НЕ удаление/kill (ложные
> срабатывания нельзя крушить автоматически). Жёсткое удаление — вручную/по подтверждению.
- [x] ⭐ **Умное обезвреживание (remediation)** (v0.24): «В карантин»/«Устранить» теперь не
      просто стирает файл, а вычищает угрозу — `monitor.remediate()`:
      (1) завершает процессы, запущенные из этого файла (terminate — иначе exe заблокирован);
      (2) снимает персистентность — убирает записи автозагрузки HKCU Run/RunOnce + папка Startup;
      (3) находит копии по SHA-256 в типичных папках (Downloads/Desktop/Temp/AppData) и
      обезвреживает их тем же действием. Подключено к кнопкам GUI и к фоновому `monitor` (CLI).
      Проверено: оригинал + копия в карантине одним кликом.
      ОГРАНИЧЕНИЯ (на потом): HKLM-автозагрузка и службы/Планировщик — нужен админ (уровень 2);
      .lnk-ярлыки в Startup не разворачиваются до цели; поиск копий поверхностный (без рекурсии).
      Честно: гарантированной чистки руткитов/инъекций user-mode НЕ даёт (этап 4 — драйвер);
      при тяжёлом заражении — офлайн-скан с загрузочной флешки + бэкапы.
- [x] `watchdog` — слежение за новыми файлами (Downloads/Desktop) → автоскан → карантин
- [x] `psutil` — снимок процессов + диф (новые процессы), пауза/возобновление

- [x] Простая эвристика: запуск из Temp/Downloads → warn
- [x] Тест на Mac: новый файл → скан → авто-карантин; снимки/диф/suspend

## Этап 3.5 — download-gatekeeper (идея пользователя, разнесена по этапам)
> Хотим: скачиваемый файл проверяется ДО завершения загрузки, и докачивается/
> становится доступным только после одобрения. Три пути реализации:
> 1. **Расширение браузера** (downloads API) — пауза загрузки до вердикта. User-mode,
>    реалистично, но только для поддержанных браузеров. → можно сделать в рамках этапа 3.
> 2. **Локальный MITM-прокси** — скан файла до отдачи браузеру. Требует вскрытия HTTPS
>    и своего сертификата. Тяжело, для учебного этапа избыточно. → отложено.
> 3. **Minifilter-драйвер** — настоящая блокировка открытия/записи файла для любой
>    программы в реальном времени. → это этап 4 (см. ниже).

> ЭТАПЫ 1–3 (всё user-mode: сканер, GUI, фоновая защита, установщик) — подпись НЕ нужна,
> всё БЕСПЛАТНО. Единственное — косметическое предупреждение SmartScreen «неизвестный
> издатель» при запуске скачанного exe → жмёшь «Выполнить в любом случае». Платный
> code-signing сертификат — опционален, только чтобы убрать предупреждение; нам не нужен.
> ПЛАТНОЕ появляется только на ЭТАПЕ 4 (драйвер ядра) и только для раздачи драйвера чужим;
> для своего ПК этап 4 тоже бесплатен (test-signing).

## Этап 4 — резидентная реал-тайм защита (kernel-mode, hardcore на C/C++)
> Эталонная архитектура «максимума»: драйвер ядра (датчик на C/C++) ловит события
> НА ЛЕТУ и спрашивает Python-«мозг» — блокировать? Это потолок защиты на Windows.
> Сначала доводим user-mode этапы 2–3 (≈80% пользы), потом беремся за драйвер.

### Что ловит драйвер ядра (Ring 0) — то, чего user-mode не может
- [ ] **Minifilter** — перехват открытия/чтения/ЗАПИСИ файлов → блок первой записи
      шифровальщика, блок запуска вредоносного exe до вердикта.
- [ ] **Process/handle callbacks** (`PsSetCreateProcessNotifyRoutine`, `ObRegisterCallbacks`)
      — блок старта процесса, защита от инъекций, запрет доступа к памяти `lsass` (кража паролей).
- [ ] **Registry callbacks** — блок записи в автозагрузку в реальном времени.
- [ ] **Драйвер-фильтр клавиатуры** — видит ввод и кто его перехватывает → ловит КЕЙЛОГГЕР.
- [ ] Канал «драйвер ↔ Python-сервис» (наш мозг остаётся на Python).

### ⚠️ Честно про подпись (важно — касается УЖЕ драйвера, не только самозащиты)
- 64-битная Windows НЕ загрузит неподписанный драйвер ядра.
- **Разработка/свой ПК — бесплатно:** test-signing mode (`bcdedit /set testsigning on`) +
  самоподписанный драйвер. Так мы и пишем/тестируем драйвер (в виртуалке!).
- **Раздача другим (без test-signing) — платно:** EV-сертификат + аттестация у Microsoft.
- **Самозащита (PPL/ELAM)** — отдельный платный уровень, сами процесс неубиваемым; вероятно
  НЕ делаем (дорого). Решение пользователя: целимся в kernel-драйвер (test-signed для себя),
  платную дистрибуцию/PPL — опционально/потом.
- ⚠️ Баг в драйвере = синий экран всей системы. Тестировать только в виртуалке.

## Стратегия защиты от активных угроз (как боремся)
> Оборона слоями. Главное оружие user-mode — МГНОВЕННАЯ ПАУЗА процесса (suspend):
> сначала заморозить, потом разбираться. Полное предотвращение первого действия — драйвер (этап 4).

- **Слой 1 (сильнейший) — до запуска:** сигнатуры VT/ClamAV/мульти-источник + download-gatekeeper.
  Известная зараза не запускается вовсе. Упор сюда.
- **Слой 2 — персистентность:** монитор автозагрузки (Run/Startup/Планировщик).
- **Слой 3 — поведение + сдерживание (user-mode):**
  - **Шифровальщики:** приманки-canary + детект по СКОРОСТИ изменения файлов → `suspend`
    → ограничиваем урон (первые файлы спасти нельзя — латентность). Бэкапы/VSS как страховка.
  - **Стилеры:** доступ к чувствительным файлам (браузер/кошельки) + сеть → suspend; фиды плохих адресов.
  - **Кейлоггеры:** ловим по сигнатуре + автозагрузке; активный хук из user-mode почти не детектится.
- **Слой 4 (этап 4) — драйвер ядра:** перехват операций на лету = истинная профилактика.
- **Честно пользователю:** ни один AV не 100%; бэкапы обязательны.

## Фишки безопасности (чем выделиться) — ✅ одобрено, разложено по этапам
> Почти всё одобрено пользователем. Метка [этап N] показывает, куда ложится.
> Большинство — фоновый режим (этап 3), user-mode, без драйвера ядра.

- [ ] **Монитор автозапуска (persistence)** [этап 3] — снимок Run-ключей реестра,
      папки «Автозагрузка», Планировщика, Служб → алерт при появлении нового
      (там прячется ~90% малвари). ⭐ ПРИОРИТЕТ. Смыкается с мониторингом процессов.
- [ ] **Ловушка от шифровальщиков (canary)** [этап 3] — несколько маленьких файлов-
      приманок на папку (килобайты, не пустые), процесс начал их массово менять →
      заморозить. Нагрузки на диск/память нет. Реальная защита от ransomware (`watchdog`).
- [ ] **Поведенческая эвристика процессов** [этап 3] — следим за поведением, а не файлом:
      документ запускает PowerShell/cmd, exe из Temp/Downloads лезет в сеть, неподписанный
      плодит процессы → «очки подозрительности» → алерт/заморозка. Лёгкая (`psutil`).
- [ ] Авто-скан флешек/USB при подключении [этап 3].
- [ ] Проверка ссылок (анти-фишинг) по URLhaus/PhishTank — до клика [этап 3].
- [ ] Монитор сетевых подключений + подсветка «плохих» адресов (ThreatFox) [этап 3].
- [ ] Контроль целостности критичных файлов (tripwire): `hosts`, DNS, системные [этап 3].
- [ ] **«Семейный режим»** — отчёты человеческим языком + авто-сканы по расписанию [этап 3].
- [ ] Приватность как фича: всё локально, наружу только хэши в выбранные базы [принцип].

### Защита самой программы (когда станет фоновой, этап 3–4)
